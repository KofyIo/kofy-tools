Los 6 Notion DBs

Notion es la base de datos. Los HTMLs solo leen y escriben vía el Cloudflare Worker que ya existe.

DB 01
Órdenes
Una fila por pedido. Encabezado del cliente, estado actual, las 5 etapas (cada una con firma + fecha), notas QC, notas tostador, número de tostados.
→ Clientes, ← Items, ← Usos
DB 02
Items de Orden
Líneas del pedido. Presentación × Tostado × Cantidad × Grano/Molido × Origen. Una orden tiene N items.
→ Órdenes, → Orígenes
DB 03
Clientes
Nombre / Razón Social, Cédula / RIF, contacto, historial (rollup de Órdenes). Re-uso: si ya existe un cliente, una nueva orden lo encuentra.
← Órdenes
DB 04
Orígenes
Un origen = un caficultor × varietal × tostado default. Caficultor, hacienda, MSNM, proceso. KG totales y remanente son rollups desde Quintales.
← Quintales, ← Items, ← Usos
DB 05
Quintales
Cada saco físico. Linked a un Origen. KG iniciales, letra (A/B/C), fecha de compra, status (activo / agotado). El remanente por quintal lo calcula la app desde Usos.
→ Orígenes, ← Usos
DB 06
Usos de Café Verde
El libro mayor. Cada vez que se aparta café verde para una orden, una fila: quintal, kg, orden, firma, fecha. Es el corazón del accountability.
→ Quintales, → Órdenes, → Miembros
DB 07 · auxiliar
Miembros
Kafay, Partner, Leo, Primo (José). Cuatro filas, fijas. Cada firma en cualquier DB apunta aquí — eso permite dashboards por persona sin escribir nombres a mano.
← firmas en Órdenes y Usos

Cómo se conecta · vista del Order ID

CLIENTE ─→ ORDEN (PETIT-3003) │ ├─→ ITEMS ─→ ORÍGENES (MG-MC-CM, etc.) │ ├─→ ESTADO 5 etapas ─→ cada una: firma → MIEMBRO │ └─→ USOS DE CAFÉ VERDE │ ├─→ QUINTAL (MG-MC-CM-2603-A) ─→ ORIGEN └─→ MIEMBRO (firma)

El ORDER ID aparece en Órdenes, en Items, y en cada fila de Usos que se haya apartado para esa orden. Ese vínculo permite preguntar a Notion: "¿de qué quintal salió cada gramo del pedido PETIT-3003, y quién lo apartó?" — y obtener la respuesta en una sola consulta.

Convenciones de ID

ORDER ID
PETIT-3003
Nombre del cliente (en mayúsculas, abreviado si hace falta) + DDMM de la fecha de pedido. Único, legible, no requiere base de datos para generar.
ID ORIGEN
MG-MC-CM
Iniciales del caficultor + iniciales del varietal + iniciales del tostado default. Tres bloques separados por guión. Si no aplica tostado default → dos bloques.
ID QUINTAL
MG-MC-CM-2603-A
ID Origen + DDMM de la compra + letra (A, B, C…) para distinguir sacos de la misma compra. Un quintal físico = una letra.
USO ID
USO-PETIT-3003-001
Auto: prefijo USO + Order ID + índice secuencial. Solo para referencia interna del libro mayor; nadie lo escribe a mano.

AN, CM y MT son los tres blends propios de Kofy. Grado A es un blend regional (combina varios orígenes de una misma zona, escogidos por calidad). Single O es origen único — una sola finca, un solo lote. Las presentaciones BOX y SACHET solo admiten Single O.

Presentaciones · enum

1 KG · 500g · 250g · BOX · SACHET

BOX y SACHET son formatos retail listos para mostrador. Los demás son bolsas tradicionales.

Inventario · regla de deducción

El sistema rastrea remanente por quintal (cada saco físico), pero las alertas de stock bajo se calculan a nivel de origen (suma de remanentes de todos los quintales activos). Esto refleja la realidad operativa: Leo trabaja con sacos individuales, pero el inventario crítico es por origen — si se acaba MG-MC-CM, se acaba para todas las órdenes que lo pidieron.

Cuando se registra un Uso, la app:

1. Resta KG del quintal seleccionado → nuevo remanente del quintal. 2. Si el quintal queda en 0 o <0.5 kg → status: agotado. 3. Suma remanentes de todos los quintales activos del mismo Origen. 4. Si esa suma cae bajo umbral (default: 10 kg) → alerta en /inventario.
