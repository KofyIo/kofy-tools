1 · Mission

Kofy's warehouse system orchestrates coffee orders through 5 stages, deducts green coffee + packaging inventory automatically, and tracks who-did-what via per-stage firmas. The missing layer is communication: the right person needs to be notified at the right moment about the action they own. And the customer needs to know when their order is shipped, ready, delayed, or cancelled.

Build a comms module that subscribes to events fired by the existing apps, dispatches notifications via email · WhatsApp · SMS, logs every send in Notion, and offers a small settings UI so the team can tune which channels carry which triggers.

Critical: the module must be fully decoupled. Existing apps emit events; the comms module listens. Existing apps know nothing about the comms module. Bring back ZERO modifications to existing files (only ADDITIVE event hooks — see §6).

Patterns to preserve

  • Persona selector — at boot, every HTML checks localStorage["kofy_warehouse_persona_v1"]. If absent, shows overlay with the 4 Miembros chips. Once picked, all firmas are auto-attributed. Comms module reads the same key.
  • Cascade firma — signing stage N marks all unsigned earlier stages with same persona/date in one PATCH.
  • Smart refresh — every app polls every 15-30s, but skips if a modal is open or focus is on an input/select/textarea.
  • Mobile-first — bottom-sheet modals, large tap targets (44px min), viewport-fit=cover, safe-area insets.
  • Notion-as-source-of-truth — no client-side cache that diverges. Polling re-reads.

5 · Comms module · what to build

5.1 · Channels & recommended providers

ChannelRecommended providerWhy
EmailResend (or Mailgun)Modern API, generous free tier, simple. Resend has 100/day free, Kofy can stay there.
WhatsAppTwilio WhatsApp APIWhatsApp Business API directly is slow approval; Twilio is faster onboarding. For Kafay's volume, Twilio works for v1.
SMSTwilio SMSSame provider as WhatsApp = one API to learn, one set of credentials.
Decision point. Before coding, ask Kafay to (a) sign up for Resend + Twilio (or alternatives), (b) get an approved WhatsApp sender ID, (c) provide a "from" email address verified on the chosen domain. Don't build assuming credentials exist — fail gracefully when they don't.

5.4 · Worker route extension

The existing worker proxies Notion. You'll add a new route that proxies notification providers (keeping API keys server-side).

POST /notify
// Body:
{
  "channel": "email" | "whatsapp" | "sms",
  "to":      "alice@example.com" | "+58412...",
  "template": "order_ready_customer",    // see 5.5
  "data":     { orderId, customerName, items, ... },
  "meta":     {                                  // for the Notion log
    trigger: "order_advanced_L",
    relatedOrden: "abc-page-id",
    triggeredBy: "leo-page-id"
  }
}

// Response:
{ status: "sent" | "failed", notifId: "NOTIF-...", error?: "..." }

Worker env vars needed: RESEND_API_KEY, TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN, TWILIO_WA_FROM, TWILIO_SMS_FROM, NOTIFICACIONES_DB_ID, FROM_EMAIL.

5.6 · Settings UI (comms.html)

Single HTML following the warehouse module patterns (persona selector at boot, dark theme, brief design tokens, smart refresh). Contains:

  • Trigger matrix — table of the 9 triggers × the 3 channels. Each cell: checkbox (enabled), recipient picker. Saved to Notion as a settings page or to localStorage for v1.
  • Recipient management — when adding/editing a Miembro or Cliente, the team can record email + WhatsApp + SMS. (Suggest adding these 3 fields to Miembros and Clientes via a small migration.)
  • Notification log viewer — last 50 notifications: trigger, channel, recipient, status, body preview, retry button if failed.
  • Test sender — pick a channel + recipient + template, fire a real notification. Critical for debugging during integration.

6 · Integration contract (plug-and-play)

The whole point of this handoff. The comms module is self-contained. The only "integration" required when bringing it back is: ensure each existing app fires a CustomEvent at the right moments. The comms module listens to those events and acts. No imports, no dependencies, no shared state.

6.1 · Event bus pattern

Existing apps fire events on the global window:

window.dispatchEvent(new CustomEvent("kofy:orderCreated", {
  detail: {
    orderId:    "PETIT-3003",
    orderPageId:"abc...",
    clientId:   "xyz...",
    clientName: "PETIT",
    items:      [{ presentacion, tostado, cantidad, ... }],
    eta:        "2026-05-20",
    createdBy:  "leo-page-id"
  }
}));

Your comms.html (or a tiny kofy-comms-loader.js) listens:

window.addEventListener("kofy:orderCreated", async (e) => {
  const { detail } = e;
  if (!userSettings.triggers.order_created.enabled) return;
  const leo = await getMemberByName("Leo");
  await sendNotification({
    channel: "whatsapp",
    to: leo.whatsapp,
    template: "order_created_leo",
    data: detail,
    meta: { trigger: "order_created", relatedOrden: detail.orderPageId, triggeredBy: detail.createdBy }
  });
});

6.3 · Bootstrap pattern

For the comms module to receive events, it must be loaded in each app. The cleanest way:

<!-- At the bottom of each warehouse-*.html, ONE line: -->
<script src="kofy-comms-loader.js"></script>

This loader is a small file (~50 lines) that:

  • Reads localStorage for comms settings (or fetches from Notion settings page)
  • Registers all event listeners
  • Calls the worker /notify route as needed
  • Logs to the Notificaciones DB

The full settings UI lives in comms.html. The lightweight loader runs in every other app to actually dispatch.

6.4 · What you DO NOT change in existing apps

Do not modify any existing file's business logic. The integration is one-line per hook point (a single window.dispatchEvent call) and the loader <script> tag. That's it. If you find yourself rewriting markStageCascade or createOrder to "make comms work", stop — that's a signal the event payload is wrong, not the integration.

8 · Testing checklist before integration

  1. Worker /notify route returns 200 with a "fake" sandbox call — no real provider yet.
  2. Notificaciones DB schema created in Notion.
  3. Settings UI loads, persona selector works, design matches.
  4. Test sender successfully sends one email + one WhatsApp + one SMS to Kafay's own contacts (real-world live test).
  5. Manually fire window.dispatchEvent(new CustomEvent("kofy:orderCreated", { detail: {...mock} })) in console — observe the loader log a notification to Notion and call /notify.
  6. Failure cases handled gracefully: no credentials → status "skipped" with reason; invalid recipient → status "failed" with provider error; rate limit → retry queue (or just log and move on).
  7. Smart refresh respected (no clobbering form input on poll).
  8. Mobile screenshot: settings UI usable on iPhone-sized viewport.

9 · Decisions to bounce back to Kafay

  1. Provider accounts: Has Kafay set up Resend + Twilio? What's the "from" email and WhatsApp sender ID?
  2. Recipient policy: Should customer notifications go to phone primarily (WhatsApp) or email primarily? Venezuelan customers heavy WhatsApp users.
  3. Cadence checker: B2B cadence (#8 trigger) needs a scheduled job (cron). Cloudflare Workers support cron triggers. Add to worker, or punt to v2?
  4. Customer portal: Kafay mentioned wanting a read-only HTML where customer sees their order status. Worth bundling with comms (the customer link in emails would point there) or separate?
  5. Templates editorial: Kafay should write the actual copy of the messages — the templates above are placeholders. Conexión-aligned voice. Spanish primary.
  6. Compliance: WhatsApp messages outside the 24h window require pre-approved templates. Plan for that?
